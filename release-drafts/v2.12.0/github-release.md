# Firecrawl v2.12.0

## Improvements

- **`search` monitor target** — Added a first-class `search` monitor target that runs SERP queries on a schedule and alerts on new or materially-developed results, alongside the existing `url` target. The target accepts `queries[]`, `searchWindow`, `includeDomains[]`, `excludeDomains[]` (exclude wins on overlap), and `maxResults`, with effective depth derived from `judgeEnabled` (`raw` returns SERP hits with no LLM calls when off, `deep` runs the LLM judge stack when on). Goal-versioned URL/event dedupe (keyed on goal + monitor name + queries), stable opaque event keys to prevent label-drift duplicates, cadence re-judging even when SERP snippets are unchanged, and a `material_dev` alert mode that re-alerts only on material developments of a known event. Exposed across the JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir SDKs with a `MonitorSearchTargetResult` type.
- **Monitor search judgment stack** — Goal-driven search results go through a verifier → criteria → skeptic → reviewer → event-resolver → run-summarizer pipeline with strict subject/topic matching and hard exclusions, so monitors with `judgeEnabled: true` get curated alerts instead of raw SERP noise. The judge runs through a shared LLM provider that prefers Vertex Gemini and falls back to Google GenAI, attaches Vertex billing labels (`functionId`, `teamId`, `monitorId`, `monitorCheckId`) on every call for traceability, and applies the same labeling to monitor diff-change judging.
- **Search monitor billing** — Search monitors now bill at a flat, deterministic rate per check: `SEARCH_CREDITS` is charged `ceil(totalResults / 10) * 2`, judged results are charged 1 credit per result (down from 5), and `raw` mode pays nothing for the judge. Deep-mode per-page scrapes flow through the shared scrape pipeline as queued jobs (billed under `CREDITS` like any other scrape) rather than being double-charged on the search check. Actuals are recorded on the check as `flatSearchTargetCredits`, and search-target pages carry 0 per-page credits. Estimates surfaced at monitor create/update include both the search call and a per-result judge allowance.
- **`menu` scrape format** — Added a new `menu` format that returns deterministic structured restaurant-menu data as `document.menu`, mirroring the existing `product` format end-to-end. Beta-gated via the `menuBeta` team flag (teams without the flag get no `menu` field and no error), with an `MENU_EXTRACTION_SERVICE_URL` self-host kill-switch: if the upstream menu engine is unreachable the scrape still succeeds with a `Menu format is not available` warning instead of failing. Exposed across the JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir SDKs, with the Rust SDK bumped from `2.9.1` to `2.10.0` since `Format::Menu` is a source-breaking addition to its public `Format` enum.
- **Hosted `/v2/parse` file uploads** — Added an authenticated two-step upload flow for parsing local files without exposing a Firecrawl API key to the model or shell. `POST /v2/parse/upload-url` mints a short-lived, team-scoped `uploadRef` plus signed upload parameters; the client uploads bytes directly to storage; and `POST /v2/parse` then accepts JSON with `uploadRef` and reuses the existing parse pipeline. Refs are HMAC-signed with a 10-minute expiry and bound to team, object path, filename, content type, and max size; production storage uses GCS signed POST policies with `content-length-range` enforcement; uploaded objects are best-effort deleted after parse. Multipart `/v2/parse` continues to work unchanged.
- **Express 5 upgrade** — Ported the API and worker to Express 5.2.1 with matching `@types/express` and `express-ws` updates, including explicit re-application of `express-ws` to the v1 and v2 routers so compiled routers expose `.ws` under Express 5.
- **TypeScript 7 RC compiler** — Switched the API build compiler to `typescript@7.0.1-rc` for roughly 3× faster clean builds (median ~1.84 s vs ~5.60 s on TS 5.9.2). The classic compiler API consumer stays pinned to TS 5 via a `typescript5` alias, `knip` runs through a TS 5 wrapper to avoid runtime API breaks, and TS 7's stricter defaults are made explicit in `tsconfig.json` to preserve the current type-checking posture.
- **NuQ FoundationDB metrics** — Added FDB queue gauges alongside the existing NuQ metrics. `nuq_fdb_queue_<queue>_job_count` reports per-status counts (`queued`, `active`, `backlog`, `completed`, `failed`), `firecrawl_nuq_fdb_pending_jobs` exposes a worker-load gauge for autoscaling, and a new `GET /admin/<BULL_AUTH_KEY>/nuq-fdb-metrics` endpoint serves them in Prometheus format.
- **`SELF_HOSTED_DOMAIN` env var** — Self-hosted instances behind a reverse proxy can now set `SELF_HOSTED_DOMAIN` to the externally-reachable host, fixing crawl status URLs, batch-scrape URLs, parse-upload links, and pagination `next` links that previously pointed at the internal bind address (e.g. `http://localhost:3002`). The resolver also honors `X-Forwarded-Host` and preserves ports; cloud behavior is unchanged when neither is set.
- **Search highlights `/highlight` service** — The `highlightsBeta` `/v2/search` path now delegates snippet rendering to the new highlight model service: the full markdown and query are POSTed to `/highlight` (under the existing service URL and token envs) and the reassembled markdown is returned as the snippet. The legacy client-side sentence-split/budget pipeline has been removed.
- **Screenshot cache cutoff** — Screenshot and full-page screenshot cache entries created before `2026-06-24T20:00:00Z` are no longer reused — affected scrapes now force a fresh capture instead of returning broken cached URLs. The cutoff is exposed as a `SCREENSHOT_INDEX_CUTOFF_MS` constant for self-host operators who want to inspect or override it.
- **`proxy: enhanced` / `stealth`** — Fixed `proxy: enhanced` and `proxy: stealth` silently falling back to non-stealth engines on the cloud — fallback selection now filters down to stealth-capable engines when stealth is requested, so the user's choice is actually honored. Self-hosted deployments without fire-engine keep working because the guard preserves the original engine list when no stealth-capable engine qualifies.
- **Autumn is the single source of truth for credits** — Removed the last ACUC-based credit accounting paths in the v0/v1/v2 hot path. The v0 `scrape`, `crawl`, and `search` controllers now check credits via `autumnService.checkCredits` (matching v1/v2), the unverified agent-key 50-credit cap reads `getTeamBalance().usage` from Autumn instead of `req.acuc.adjusted_credits_used`, and the dead `checkTeamCredits` / `supaCheckTeamCredits` / `evaluateTeamCredits` helpers have been removed. Behavior is preserved: 402 on insufficient credits, fail-open when Autumn is unavailable or self-hosted. v0 crawl no longer clamps `crawlerOptions.limit` to ACUC `remainingCredits` when Autumn allows the request (including overage), matching v1/v2.
- **ACUC concurrency v48** — Pointed the API's `auth_credit_usage_chunk` RPCs at `v48`, which makes single-team orgs resolve concurrency as `GREATEST(override, plan)` (stops stale coupon overrides throttling paid teams) and gives pay-as-you-go orgs a concurrency floor of 5 instead of 2.
- **OpenAI fallback for scrape query answers** — `performFreeformQuery` now falls back to `gpt-4o-mini` between Google Gemini and Vertex Gemini, so freeform scrape queries (`format: "question"` / `query`) still return `document.answer` in OpenAI-only environments when Gemini is throttled and Vertex credentials are missing.
- **Search credit double-tracking fix** — Search credits are now tracked to Autumn only at request time via `billTeam` / `billScrapeJob`. `processBillingBatch` only commits the ledger via `bill_team_6` and refunds request-tracked usage on failure, eliminating a path that could double-count search credits when both the request and the batch tracked the same usage.
- **Team-scoped historical credit and token usage** — `/v1/team/credit-usage/historical`, `/v2/team/credit-usage/historical`, and the matching `token-usage/historical` endpoints now return only usage tracked against the API key's team. The previous customer-level fallback that fired when the team's Autumn entity lookup 404'd is gone; teams with no entity yet correctly return an empty history.
- **`autumn-js` 1.2.30** — Bumped `autumn-js` from `1.2.13` to `1.2.30` in `apps/api`, dropping the `AUTUMN_REQUEST_TRACK_EXPERIMENT` and `AUTUMN_REQUEST_TRACK_EXPERIMENT_PERCENT` gates entirely so credits are always tracked through the request-scoped path.

## Fixes

- Resolved multiple CVEs across the API and supporting workspaces by upgrading `multer` to `2.2.0` in `apps/api` (GHSA-3p4h-7m6x-2hcm, GHSA-72gw-mp4g-v24j), `undici` to `7.28.0` with a pnpm override locking it major-bound in `apps/api` and `apps/test-suite` (GHSA-35p6-xmwp-9g52, GHSA-g8m3-5g58-fq7m, GHSA-hm92-r4w5-c3mj, GHSA-p88m-4jfj-68fv, GHSA-pr7r-676h-xcf6, GHSA-vmh5-mc38-953g, GHSA-vxpw-j846-p89q), and `artillery` to `^2.0.33` in `apps/test-suite`.
- Removed the malicious `serpapi-python==0.1.5` pin from the `gpt-4.1-company-researcher` example in favor of the official `google-search-results` package (which provides the same `serpapi.google_search.GoogleSearch` import), and dropped the unavailable `firecrawl==0.1.2` pin that was blocking installs.
- Fixed Playwright-engine scrapes losing the `Cookie` header on 302 redirects (including cross-subdomain hops). The cookie is now parsed and seeded into the browser context via `context.addCookies()`, scoped to the registrable domain, so authenticated pages that redirect before rendering no longer come back as the login page.
- Fixed `redactPII` and change-tracking diffs causing scrape-worker pods to shell out to `diff` for every page — the API now uses an in-process diff implementation, removing a fork-per-scrape and a hard dependency on `diff` being installed in the container.
- Fixed external `$ref` passthrough in `/v2/extract` schemas by replacing `json-schema-ref-parser` with an internal-only `$ref` resolver that refuses non-internal references, fixing a class of SSRF-shaped errors where extract requests with external `$ref` URLs could cause the worker to fetch arbitrary HTTP endpoints.
- Fixed v0 FoundationDB-backed crawl status reads and result blobs being routed to the wrong backend, so v0 crawl polling no longer returns stale or missing data on FDB clusters; FDB `waitForJob` now honors the caller-supplied timeout instead of clipping to the optional-mode wait.
- Fixed single NuQ/Postgres inserts targeting the backlog table being silently promoted to `queued` — the backlog status is now preserved end-to-end, so backlog-only queues behave correctly under the Postgres backend.
- Fixed the FoundationDB native binding failing to build under pnpm by allowlisting the `foundationdb` package build script in both `apps/api/package.json` and `apps/api/pnpm-workspace.yaml`, restoring linux/arm64 image builds that include the native binary.
- Fixed `proxy: enhanced` and `proxy: stealth` scrapes silently routing through non-stealth engines on the cloud — the fallback list is now restricted to stealth-capable engines when stealth is requested, with a guard to preserve the original list on self-hosted deployments without fire-engine.
- Fixed v2 `skipTlsVerification` default drift across request validation, scrape metadata, and engine feature flags by centralizing the default in `applyScrapeOptionsDefaults`. Omitted values now consistently default to bypassing TLS verification; explicit `false` and the existing headers/actions safety exception are preserved.
- Fixed search credits being double-counted when both the request-scoped tracker and the billing batch tracked the same usage — `processBillingBatch` no longer calls Autumn `trackCredits` and refunds request-tracked usage on errors via `refundRequestTrackedCredits`.
- Fixed `/v1` and `/v2` historical credit/token usage endpoints leaking other teams' usage in multi-team orgs when the team's Autumn entity 404'd — the customer-level fallback has been removed and missing entities correctly return an empty history.
- Fixed monitor search billing reading as zero when `searchCompleted` had not yet flushed — completion now waits for `searchCompleted` so search-check actuals are never recorded as zero, and the judged-result count stays correct under retries.
- Fixed `judgeEnabled: false` on search monitors still running LLM judge calls — the flag now actually gates the judge, so raw mode produces no LLM calls or judge credits.
- Fixed the monitor judge over-flagging adjacent or "credible-source" results as meaningful — the judge now requires a specific subject/topic match, cutting false positives in goal-driven monitors.
- Fixed search-monitor checks silently returning "no changes" on transient empty or rate-limited SERP responses — the runner now retries transient failures and surfaces real search errors instead of hiding them as a clean run.
- Fixed self-hosted snip tests failing with `fetch failed` whenever `IDMUX_URL` was unreachable — self-hosted runs now fall back to the configured `TEST_API_KEY` / `TEST_TEAM_ID` with a warning, while production runs stay strict.
- Fixed Python SDK imports emitting Pydantic `UserWarning` about `MonitorPageSnapshot.json` and `MonitorPageDiff.json` shadowing `BaseModel.json()` — both classes now ship with the same import-time warning filter already used for `ScrapeFormats` and `Document`.
- Fixed `/v2/search/research` requests not logging the originating client because the `Origin` header was dropped before reaching the research proxy — the controller now forwards and logs `Origin` so research traffic is attributable.
- Patched the `map` redirect snip and the FDB router `waitForJob` regression test to no longer depend on external sitemap discovery or a live Redis instance, so the test suite is stable in the fetch-only and Redis-down CI lanes.

## API

- Added `POST /v2/parse/upload-url`, which mints a short-lived team-scoped `uploadRef` plus signed storage upload parameters. Authenticated only — keyless callers cannot mint upload refs. Refs are HMAC-signed and expire after 10 minutes, bound to the team, object path, filename, content type, and max size. New self-host env vars: `PARSE_UPLOAD_REF_SECRET` (required outside local/dev/test), `GCS_PARSE_UPLOAD_BUCKET_NAME`, and `PARSE_UPLOAD_STORAGE_DRIVER=gcs`.
- Added an `uploadRef` field to `POST /v2/parse` as an alternative to multipart uploads. The parse-by-ref request must authenticate as the same team that minted the ref; the existing multipart `/v2/parse` behavior is unchanged.
- Added `"menu"` to the accepted values of `formats` on `POST /v2/scrape` and the endpoints that share its options, returning `document.menu` (canonical `MenuProfile` shape). Gated behind the `menuBeta` team flag and the `MENU_EXTRACTION_SERVICE_URL` self-host env var; teams without access get no `menu` field and no error.
- Added a `search` monitor target on `POST /v2/monitor` and `PATCH /v2/monitor/:id` accepting `queries: string[]`, `searchWindow`, `includeDomains?: string[]`, `excludeDomains?: string[]`, and `maxResults`. `goal` is required only when `judgeEnabled` is true; partial updates validate the merged state. Client-side `depth`, `alertMode`, and `recheckAfter` are now internal-only and stripped on create/update (effective `depth` derives from `judgeEnabled`); `scrapeOptions` is ignored on search targets. Target signatures include `includeDomains` / `excludeDomains`, `maxResults`, and effective `depth` for dedupe.
- Added a `MonitorSearchTargetResult` type and a `searchTarget` block on monitor check responses for `GET /v2/monitor/:id/checks/:checkId`, including `matchedQueries[]`, event keys, judgment, `meaningfulChange`, and a `judgeDegraded` flag set when the judge path fails so callers can distinguish a degraded run from a clean one. Typed across the JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir SDKs.
- Added `material_dev` to the monitor `alertMode` enum on search targets, which re-alerts only on material developments of a previously-alerted event (versus `first_match`, which alerts once per stable event key).
- Changed search monitor billing to a flat per-check rate: `SEARCH_CREDITS` is charged `ceil(totalResults / 10) * 2`, judge calls are billed at 1 credit per judged result (down from 5), `raw` mode incurs 0 judge credits. Per-page scrapes triggered by deep-mode search flow through the shared scrape queue and bill against `CREDITS` like any other scrape; the actual flat cost is recorded on the check as `flatSearchTargetCredits`. LLM token usage on search judging is recorded for observability only.
- Added the `search` target type to the monitor SDK surface in JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir clients, with camelCase wire fields and a `MonitorSearchTargetResult` per-page result type.
- Changed the `highlightsBeta` snippet rendering path in `/v2/search` to send full markdown and the user query to the new `/highlight` model service and return its reassembled markdown as the snippet; the previous sentence-split/budget pipeline (`highlight-budget`) has been removed.
- Added a `GET /admin/<BULL_AUTH_KEY>/nuq-fdb-metrics` admin endpoint exposing `nuq_fdb_queue_<queue>_job_count` (per status: `queued`, `active`, `backlog`, `completed`, `failed`) and `firecrawl_nuq_fdb_pending_jobs` for FDB-based queue autoscaling.
- Added the `SELF_HOSTED_DOMAIN` environment variable. When set, response URLs and pagination `next` links emitted by `/v1/crawl`, `/v1/batch/scrape`, `/v1/crawl/{id}`, `/v2/crawl`, `/v2/batch/scrape`, `/v2/crawl/{id}`, `/v2/monitor` checks, and `/v2/parse/upload-url` use the configured host instead of `req.get("host")`. Resolution order is `SELF_HOSTED_DOMAIN` → first `X-Forwarded-Host` → `req.get("host")`; the value is normalized to a bare `host[:port]` even if operators include a scheme or path.
- Normalized the `skipTlsVerification` default on `POST /v2/scrape` (and endpoints that share its options) across request parsing, queued scrape metadata, and engine feature flags. Omitted values consistently default to bypassing TLS verification, explicit `false` is preserved, and custom headers/actions still default to enforcing TLS.
- Added a `SCREENSHOT_INDEX_CUTOFF_MS` constant (default: `2026-06-24T20:00:00Z`) that is applied to both the in-memory and DB cache filters so screenshot index entries created before the cutoff are skipped and the worker re-captures. Self-hosters who do not want the cutoff can override it.
- Removed the `AUTUMN_REQUEST_TRACK_EXPERIMENT` and `AUTUMN_REQUEST_TRACK_EXPERIMENT_PERCENT` environment variables. Credits are now always tracked through the request-scoped path; remove the variables from your deployment configuration.
- Removed the v0 credit-check ACUC fallback. v0 `scrape`, `crawl`, and `search` now check credits exclusively via Autumn — same gate as v1/v2 (402 on insufficient credits, fail-open on Autumn-down or self-host). v0 crawl no longer clamps `crawlerOptions.limit` to ACUC `remainingCredits` when Autumn allows the request, matching v1/v2.
- Migrated the unverified agent-key 50-credit cap to read `getTeamBalance().usage` from Autumn instead of `req.acuc.adjusted_credits_used`. The cap responds with 402 `unverified_credit_limit_reached` and `credits_used` sourced from Autumn; transient Autumn lookup failures skip the cap (fail-open), matching the main credit check.
- Changed the API's `authCreditUsageChunk` / `authCreditUsageChunkFromTeam` SQL RPCs to call `auth_credit_usage_chunk_48` / `_48_from_team`, activating the concurrency fixes (single-team orgs use `GREATEST(override, plan)`; pay-as-you-go orgs use a concurrency floor of 5 instead of 2).
